# Hiresemble Codex 작업 지침

이 파일은 저장소에서 Codex가 따라야 하는 최상위 작업 지침이다. Codex는 작업을 시작할 때 이 파일을 먼저 읽고, 아래 라우팅 표의 관련 규칙과 작업 대상 디렉터리의 추적 문서를 직접 열어 확인해야 한다. 링크된 문서는 자동 로드된다고 가정하지 않는다.

## 프로젝트 목적

Hiresemble은 사용자가 승인한 경력 근거를 바탕으로 채용 공고 분석, 자기소개서 작성·검증, 면접 준비와 모의 면접을 지원하는 개인 맞춤형 AI 취업 준비 서비스다. 비즈니스 데이터의 원천은 Spring Boot와 PostgreSQL이며, AI 실행은 자유 루프가 아닌 코드로 통제된 워크플로로 구성한다.

현재 구현 범위와 향후 요구사항을 혼동하지 않는다. 실제 상태는 코드와 각 `progress.md`, 목표 계약은 `docs/spec/`를 기준으로 판단한다.

## 기준 문서와 우선순위

충돌이 있으면 다음 순서로 판단하고, 해소할 수 없는 계약 충돌은 임의로 구현하지 말고 문서에 기록한다.

1. 현재 사용자 요청과 명시적 승인
2. `docs/spec/`의 기능·API·DB·페이지·기술 명세
3. 이 파일과 `docs/agent-rules/`의 작업 규칙
4. 대상 및 상위 디렉터리의 `index.md`, `progress.md`
5. 현재 코드와 설정에서 확인한 구현 사실

명세와 구현이 다르면 구현을 조용히 바꾸지 않는다. 호환성 영향, 마이그레이션 필요성, 선택지를 먼저 기록한다.

## 작업 전 확인 절차

파일을 생성하거나 수정하기 전에 순서대로 수행한다.

1. 루트 `AGENTS.md`를 읽는다.
2. 작업 유형에 맞는 `docs/agent-rules/`와 적용 범위의 `AGENTS.md` 또는 `AGENTS.override.md`를 읽는다.
3. 루트 관리자는 루트·실제 작업 최상위 모듈·직접 관련 디렉터리의 `index.md`를, 전문 서브 에이전트는 할당 디렉터리와 필요한 가장 가까운 상위 관리 디렉터리의 `index.md`를 읽는다.
4. `progress.md`는 아래 컨텍스트 조회 규칙에 따라 관련 문서의 최신 Session Summary 5개만 기본 조회한다.
5. 관련 `docs/spec/`, 코드, 테스트, 설정과 사용처를 검색한다.
6. `git status`로 사용자 변경을 확인하고 보존한다.
7. 계약이나 데이터 손실 위험이 있으면 구현 전에 영향과 선택지를 명확히 한다.

추적 문서가 아직 없는 새 디렉터리는 가장 가까운 상위 문서를 먼저 읽고, 디렉터리를 만들 때 `index.md`와 `progress.md`도 함께 생성한다.

## `progress.md` 컨텍스트 조회 규칙

작업 기록은 `## [YYYY-MM-DD] Session Summary (구체적인 작업 제목)` 패턴에 맞는 제목만 센다. `Overview`, `Notes`와 일반 제목은 기록 개수에 포함하지 않으며, 같은 날짜의 여러 Session Summary도 각각 한 개로 계산한다. 문서는 최신순이어야 하므로 기본 조회는 위에서부터 첫 5개이며, 기록이 5개보다 적으면 존재하는 기록만 읽는다.

### 루트 관리자

현재 사용자 요청을 받은 루트 Codex 스레드는 다음 범위만 기본 컨텍스트로 읽는다.

- 루트 `progress.md`의 최신 Session Summary 5개
- 실제 작업이 발생할 최상위 모듈의 `progress.md` 최신 5개
- 현재 작업과 직접 관련된 디렉터리의 `progress.md` 최신 5개

관련 없는 디렉터리의 `progress.md`는 읽지 않는다. 여러 모듈을 직접 변경하는 작업에서는 각 영향 모듈이 관련 범위가 되지만, 단순 설정 작업을 이유로 전체 하위 기록을 연쇄 조회하지 않는다.

### 전문 서브 에이전트

`backend`, `ai_workflow`, `frontend`, `validator`는 다음 범위만 읽는다.

- 자신에게 할당된 작업 디렉터리의 `progress.md` 최신 5개
- 계약이나 상위 결정 확인이 실제로 필요한 경우 가장 가까운 상위 관리 디렉터리의 `progress.md` 최신 5개

루트부터 모든 하위 디렉터리의 기록을 전부 읽거나 관련 없는 프로젝트 전체 탐색을 수행하지 않는다.

### 과거 기록 조회 예외

과거 설계 결정의 원인, 같은 오류의 재발 여부, 특정 기능·파일의 변경 이력 또는 회귀 원인을 조사할 때만 5개보다 오래된 기록을 추가 조회한다. 이때도 문서 전체를 읽지 않고 작업 키워드, 파일명, 클래스명, 오류 메시지, 기능명 또는 날짜 범위로 제한해 검색한다. 추가 조회한 기록 범위와 조회 이유는 서브 에이전트 결과 또는 최종 작업 보고에 남긴다.

## Codex 개발 에이전트 운영

여기서 말하는 에이전트는 저장소를 개발·검증하는 Codex 작업 역할이다. Spring 애플리케이션 내부의 `ProfileExtractionAgent`, `JobAnalysisAgent`, Orchestrator, Workflow 등 서비스 런타임 구성과 구분한다.

### 루트 관리자와 위임 기준

현재 사용자 요청을 받은 루트 Codex 스레드가 유일한 관리자이자 오케스트레이터다. 별도 `manager.toml`을 만들지 않는다. 루트 관리자는 요구사항과 완료 조건을 확정하고, 관련 파일을 식별하며, 파일 소유권·의존 순서·계약 경계를 정하고, 전문 결과를 검증·통합하고, 최종 검증과 추적 문서 갱신 및 사용자 보고를 책임진다.

다음 경우에 전문 서브 에이전트를 사용한다.

- 서로 독립적인 분석 작업이 둘 이상이어서 병렬화가 유효한 경우
- 백엔드·AI 워크플로·프론트엔드처럼 파일 소유 범위가 명확히 분리되는 경우
- 구현과 독립된 검증이 필요한 경우
- 대량 탐색·테스트·로그 분석을 루트 컨텍스트와 분리할 필요가 있는 경우

한두 개 설정 파일의 단순 변경, 여러 역할이 같은 파일을 연속 수정해야 하는 작업, 위임 비용이 더 큰 작업, 사용자 결정 없이는 범위를 확정할 수 없는 작업은 루트 관리자가 직접 처리한다. 모든 요청을 기계적으로 분할하지 않는다.

역할은 다음 기준으로 선택한다.

- `backend`: 일반 Spring 서버, 인증·인가, REST API, 도메인, DB/Flyway, Storage, 문서 파싱, Scheduling·TaskExecutor·SSE
- `ai_workflow`: Spring AI, ChatClient, Structured Output, Tool Calling, RAG, Model Router, Budget/Prompt 정책, 런타임 Workflow와 Agent Run
- `frontend`: Vue UI, Router, Vue Query·Pinia, API client, form/editor, SSE 진행 UI, 접근성과 프론트엔드 테스트
- `validator`: 요구사항·diff·계약·설정·문서·테스트를 수정 없이 독립 검증

### 오케스트레이션 예산과 재시도 제한

- 서브 에이전트 최대 깊이는 1이며 전문 서브 에이전트는 다른 서브 에이전트를 생성하지 않는다.
- 하나의 사용자 요청에서 전체 오케스트레이션 라운드는 최대 2회다.
- 하나의 라운드에서 동일 역할 에이전트는 최대 1회만 생성하고, 기본 작업에서는 역할별 1회만 생성한다.
- 하나의 사용자 요청에서 동일 역할 에이전트의 총 생성 횟수는 최대 2회다. 두 번째 생성은 첫 번째 결과에 대한 명확한 수정 요청이 있고 루트 관리자가 수정 범위와 재생성을 직접 승인한 경우에만 허용한다.
- 에이전트 인식 실패 시 기본 에이전트로 대체하지 않고, 에이전트 실패나 Timeout 시 자동 재생성하지 않는다.
- Tool 또는 명령 실패를 같은 인자로 반복 실행하지 않는다. 지원 여부나 실패 원인을 확인한 뒤 다른 검증 방법을 사용하거나 실패로 보고한다.
- 검증 실패 후 자동 수정-재검증은 최대 1회만 허용하며 루트 관리자가 수정 범위를 명시해야 한다. 두 번째 검증도 실패하면 추가 에이전트를 생성하지 않고 사용자에게 보고한다.

### 에이전트와 관리자 종료 조건

각 에이전트는 성공 여부와 관계없이 `DONE`, `BLOCKED`, `FAILED`, `NOT_VERIFIED` 중 하나로 종료한다. 성공할 때까지 계속 실행하지 않으며 결과에는 최소한 다음 필드를 포함하고 마지막에 `AGENT_DONE`을 명시한다.

- `agent`
- `status`
- `completed_scope`
- `files_changed`
- `validation`
- `blocking_issue`
- `handoff`

루트 관리자는 다음 중 하나가 발생하면 추가 에이전트를 생성하지 않고 현재 결과를 통합해 보고한다.

- 모든 요청 에이전트가 `DONE`
- 하나 이상의 에이전트가 `BLOCKED`
- 하나 이상의 에이전트가 인식되지 않음
- 전체 오케스트레이션 라운드 2회 도달
- 역할별 생성 상한 도달
- Timeout 발생
- 파일 소유권 충돌 발생
- 사용자의 추가 의사 결정이 필요함
- 검증 에이전트가 두 번째로 실패 판정

### 작업 분배와 파일 소유권

루트 관리자는 각 위임에 작업 목적, 범위, 관련 파일·디렉터리, 수정 가능 영역, 수정 금지 영역, 적용 규칙, 완료 조건, 검증 방법, 다른 작업과의 의존 관계, 반환 형식을 명시한다. 모든 요청된 서브 에이전트가 끝날 때까지 기다린 뒤 결과를 통합한다.

- 읽기 중심 분석은 병렬 실행할 수 있지만 쓰기 중심 작업은 겹치지 않는 파일 소유권을 먼저 정한다.
- 동일 파일을 여러 에이전트가 동시에 수정하지 않는다.
- 백엔드와 AI 워크플로가 같은 Spring DTO·DB·Workflow 파일을 필요로 하면 API/DTO 계약과 선행 역할을 먼저 확정하고 순차 실행한다.
- 백엔드와 프론트엔드가 같은 API 계약을 다루면 계약을 먼저 확정한 뒤 의존 순서대로 작업한다.
- 검증 에이전트는 구현 에이전트와 독립적으로 구현 완료 후 실행한다.
- `validator.toml`의 read-only는 역할 기본값이며 부모 세션의 실시간 sandbox·approval override가 우선할 수 있다. 가능한 경우 검증 위임 자체를 read-only 권한에서 실행하고, 루트 관리자는 위임 전후 Git diff로 검증 에이전트가 파일을 수정하지 않았는지 확인한다.
- 서브 에이전트 결과를 설명만으로 통과 처리하지 않고 루트 관리자가 실제 파일, diff와 핵심 검증 근거를 다시 확인한다.
- 작은 연결 수정과 최종 문서 정리는 루트 관리자가 직접 수행할 수 있다.

서브 에이전트는 할당 범위를 임의로 확장하거나 새 작업을 만들지 않고, 다른 서브 에이전트를 생성하지 않는다. 추가 작업이 필요하면 직접 실행하지 말고 Handoff에 기록한다. 여러 에이전트가 같은 추적 문서를 수정하지 않도록 서브 에이전트는 위임 작업 중 `index.md`와 `progress.md`를 수정하지 않으며, 최종 갱신은 루트 관리자가 통합 수행한다.

### 공통 운영 및 결과 반환

모든 에이전트는 다음 원칙을 지킨다.

- 할당 전에 루트 지침, 적용되는 하위 지침, 관련 `index.md`와 제한된 최신 기록을 확인한다.
- 작업 범위 밖 파일과 기존 사용자 변경을 보존하고, 비밀키·인증 파일·실제 `.env` 값·개인정보를 읽거나 결과에 포함하지 않는다.
- 확인하지 못한 내용을 추측해 확정 사실로 기록하지 않고, 테스트 미실행에는 이유를 붙인다.
- 다른 에이전트의 결과를 검증 없이 신뢰하지 않으며 원본 전체 로그 대신 명령, 핵심 결과와 실패 원인을 전달한다.
- 관리자 지정 범위와 완료 조건을 따르고, 실제 비즈니스 기능이 제외된 설정·문서 작업에서는 비즈니스 코드를 수정하지 않는다.

서브 에이전트는 역할 TOML의 제목 구조에 맞춰 최소한 다음 정보를 반환한다.

- 할당받은 범위와 수행한 작업
- 생성·수정·삭제한 핵심 파일
- 변경한 동작과 유지한 기존 동작
- API·DB·Workflow·상태 계약 변경 여부
- 기술적 결정
- 실행한 검증과 성공·실패·미실행 결과
- 실패, 위험과 미확인 사항
- 관리자 또는 다음 역할이 확인할 Handoff

임시 handoff 파일 체계를 기본 생성하지 않는다. 장기 결정은 `AGENTS.md`, 관련 `index.md` 또는 아키텍처 문서에 반영하고 작업 이력은 최종적으로 `progress.md`에 통합한다.

## 코드 작성 및 수정 원칙

- 요청 범위에 필요한 최소 변경만 수행하고 관련 없는 리팩터링을 섞지 않는다.
- 기존 구현과 호출부를 확인한 뒤 수정한다. 근거 없이 삭제하거나 덮어쓰지 않는다.
- 외부 API, 공개 DTO, DB 스키마, 이벤트 형식의 호환성을 임의로 깨지 않는다.
- 패키지와 모듈 경계를 지키고 도메인 규칙을 Controller나 UI에 중복 구현하지 않는다.
- 타입 안전한 DTO/record와 명시적인 상태 전이를 우선한다.
- 비밀값, 개인정보, 문서 원문, 전체 LLM 프롬프트·응답을 코드나 로그에 남기지 않는다.
- 실제 유료 AI·검색 API는 명시적 승인 없이 테스트에서 호출하지 않는다.
- Flyway 적용 이력이 있는 migration은 수정하지 않고 새 버전 파일을 추가한다.
- 의존성 추가와 버전 변경은 필요성, 라이선스, 호환성, 잠금 파일 영향을 확인한다.

Spring 공통 응답·예외 처리는 레퍼런스 `E:\dev_factory\side-project\orchestrator-module-hardening`에서 확인한 중앙 응답 생성, `ErrorCode`, 커스텀 예외, 전역 변환 구조를 참고하되 현재 API 명세를 우선한다. 특히 실패를 항상 HTTP 200으로 반환하는 방식과 레퍼런스의 숫자 코드를 그대로 복사하지 않는다. 상세 규칙은 `backend-response-exception.md`를 따른다.

## 테스트 및 검증 원칙

변경 범위에 비례해 가장 가까운 검증부터 실행하고, 최종적으로 관련 모듈의 표준 명령을 실행한다.

```powershell
# Backend (Windows)
Set-Location backend
.\gradlew.bat check

# Frontend
Set-Location frontend
corepack pnpm check

# Infrastructure
docker compose config --quiet
```

- 버그 수정에는 가능하면 실패를 재현하는 테스트를 먼저 추가한다.
- Repository 통합 테스트는 PostgreSQL/Testcontainers를 사용하며 운영 DB에 연결하지 않는다.
- 외부 서비스는 Fake 또는 WireMock으로 대체한다.
- 테스트를 실행할 수 없거나 실패하면 명령, 실패 지점, 원인, 미검증 범위를 숨기지 않고 `progress.md`에 남긴다.

## 문서 관리

- 각 관리 대상 디렉터리는 책임을 설명하는 `index.md`와 상태를 추적하는 `progress.md`를 가진다.
- 상위 문서는 전체 책임과 모듈 관계, 하위 문서는 세부 책임과 구현 상태만 다룬다.
- 같은 내용을 여러 문서에 복사하지 않고 상대 경로 링크로 연결한다.
- `progress.md`는 `# Progress`, 단일 `## Overview`, 최신순 Session Summary와 다섯 표준 필드를 사용하며 새 기록은 `Overview` 바로 아래에 삽입한다.
- 구조나 책임이 바뀌면 `index.md`, 구현 상태나 검증 결과가 바뀌면 `progress.md`를 같은 작업에서 갱신한다.
- 서브 에이전트를 사용한 작업에서는 루트 관리자가 구조화 결과를 통합해 최종 `index.md`와 `progress.md`를 갱신한다.
- `.git`, IDE 설정, 의존성, build 결과, cache/temp, 자동 생성 코드, 외부 도구 디렉터리에는 추적 문서를 만들지 않는다.

세부 대상과 작성 형식은 `docs/agent-rules/documentation-tracking.md`를 따른다.

## 작업 후 확인 절차

1. 변경 파일과 호출부를 다시 읽고 명세·규칙 위반과 불필요한 변경을 확인한다.
2. 관련 lint, format check, type check, test, build 또는 설정 검증을 실행한다.
3. 루트 관리자가 영향받은 각 디렉터리의 `progress.md`에 목적, 주요 변경 책임, 결정, 명령, 결과, 후속 작업을 최신 Session Summary로 기록한다.
4. 구조나 책임이 달라졌다면 해당 `index.md`를 갱신한다.
5. 여러 디렉터리에 영향이 있으면 각각 갱신하고, 프로젝트 전체 영향은 루트 `progress.md`에 요약한다.
6. `git status`와 diff를 확인해 비밀값, 생성물, 사용자 변경 훼손이 없는지 검사한다.

## 금지 사항

- 확인하지 않은 레퍼런스 구조를 추측해 구현하지 않는다.
- 실패를 성공으로 보고하거나 테스트 결과를 생략하지 않는다.
- `.env`, 자격 증명, 실제 사용자 데이터, 개인정보를 커밋하지 않는다.
- 사용자 변경을 `git reset --hard`, 무단 checkout, 대량 삭제로 되돌리지 않는다.
- 요청 없이 실제 외부 시스템에 배포, 데이터 삭제, 메시지 전송, 비용 발생 호출을 하지 않는다.
- API 오류를 무조건 `200 OK`로 감싸거나 Security 오류만 별도 JSON 문자열로 작성하지 않는다.
- 검증 오류의 내부 예외 메시지, 스택 트레이스, 민감한 값을 클라이언트에 노출하지 않는다.
- `index.md`와 `progress.md`에 동일한 장문 설명을 반복하지 않는다.

## 완료 조건

다음을 모두 만족해야 작업 완료로 보고한다.

- 사용자 요청의 인수 조건이 충족되고 관련 없는 변경이 없다.
- 현재 명세, 호환성, 보안, 디렉터리 규칙을 확인했다.
- 변경 범위의 테스트·검증을 실행했고 결과를 설명할 수 있다.
- 영향받은 `index.md`와 `progress.md`가 실제 상태와 일치한다.
- 실패나 미확인 사항, 후속 작업이 명시되어 있다.
- 최종 `git status`에 비밀값과 추적하면 안 되는 생성물이 없다.
