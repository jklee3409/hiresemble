# 공통 개발 작업 절차

## 적용 범위

코드, 설정, 문서, 테스트, CI 파일의 생성·수정·삭제와 리뷰에 적용한다. 루트 [`AGENTS.md`](../../AGENTS.md)의 공통 지침을 반복하지 않고 실제 수행 순서를 구체화한다.

## 작업 전

1. `AGENTS.md`와 이 문서를 읽는다.
2. 작업 유형별 규칙을 추가로 읽는다.
3. 역할에 필요한 루트·모듈·직접 대상 `index.md`를 읽고 책임 경계를 확인한다.
4. `progress.md`는 [`AGENTS.md`](../../AGENTS.md)의 역할별 범위에서 제목 패턴에 맞는 최신 Session Summary 5개만 읽는다. 과거 원인·오류·회귀 조사가 필요하면 키워드·파일명·클래스명·오류 메시지·기능명·날짜 범위로 제한해 추가 검색하고 이유를 보고한다.
5. 관련 명세와 코드의 정의·사용처·테스트를 `rg`로 찾는다.
6. `git status --short`와 필요한 diff를 확인한다. 커밋이 없는 저장소에서는 untracked 파일도 기존 사용자 파일일 수 있다.
7. 요청 범위, 완료 조건, 호환성 및 데이터 영향을 짧게 정리한다.
8. 여러 독립 영역을 병렬 작업할 때 파일 소유 범위를 분리해 충돌을 막는다. 서브 에이전트는 추적 문서를 수정하지 않고 루트 관리자가 최종 통합한다.

문서 계층을 처음 만드는 경우처럼 선행 문서가 실제로 없으면 가장 가까운 기존 상위 문서를 읽고, 생성 이유를 루트 `progress.md`에 기록한다.

## 구현 원칙

- 변경은 요구사항을 만족하는 가장 작은 논리 단위로 제한한다.
- 기존 코드가 있으면 삭제나 대체 전에 정의, 호출부, 테스트, 외부 계약을 확인한다.
- 패턴의 일관성보다 공개 계약과 데이터 안전성을 우선한다.
- 새 추상화는 두 개 이상의 실제 사용처나 명확한 계약 경계가 있을 때 도입한다.
- 오류를 숨기는 fallback, 광범위한 catch, 무의미한 null 반환을 추가하지 않는다.
- 임시 우회가 불가피하면 이유, 제거 조건, 테스트 공백을 `progress.md`에 남긴다.
- formatter가 관리하는 대량 변경과 실제 의미 변경은 가능한 한 분리한다.

## 검증 선택

| 변경 영역           | 최소 검증                           | 추가 검증 조건                                   |
| ------------------- | ----------------------------------- | ------------------------------------------------ |
| Markdown/링크       | 필수 섹션, 상대 링크, format 검사   | 구조 변경 시 관리 문서 형식·최신순 전체 검사     |
| Backend Java/Gradle | `.\gradlew.bat check`               | DB 연동 시 Testcontainers, API 변경 시 MockMvc   |
| Frontend            | `corepack pnpm check`               | 주요 사용자 흐름 변경 시 Playwright              |
| Compose/환경 변수   | `docker compose config --quiet`     | 서비스 변경 시 health와 실제 연결 확인           |
| Flyway              | backend check와 migration 목록 확인 | 새 migration은 빈 DB와 기존 DB upgrade 모두 확인 |
| GitHub Actions      | YAML과 로컬 대응 명령 확인          | 가능하면 PR check 결과 확인                      |

운영 데이터나 외부 유료 API에 의존하는 검증은 실행하지 않는다. 실행할 수 없는 검증은 대체하지 않은 채 성공이라고 보고하지 말고 미검증으로 기록한다.

## 작업 후

1. 변경한 파일을 다시 읽고 `git diff --check`로 whitespace 오류를 확인한다.
2. 영향 범위의 표준 검증을 실행한다.
3. 새 파일, 생성물, 비밀값이 잘못 추적되지 않는지 `git status --short --ignored`로 확인한다.
4. 루트 관리자가 각 영향 디렉터리의 `progress.md`에 다음을 구체적으로 기록한다. 서브 에이전트를 사용하지 않은 단일 작업에서는 현재 루트 스레드가 같은 책임을 갖는다.
   - 작업 목적
   - 생성·수정·삭제한 주요 파일
   - 핵심 구현과 결정
   - 실행한 명령과 결과
   - 남은 문제와 후속 작업

   기록은 KST 날짜의 별도 Session Summary로 `Overview` 바로 아래에 삽입하며 같은 내용을 여러 계층에 그대로 복사하지 않는다.

5. 구조·책임·의존 관계가 달라진 디렉터리의 `index.md`를 갱신한다.
6. 전체 프로젝트 영향이 있으면 루트 `progress.md`에 요약한다.
7. 최종 보고에는 검증 실패와 확인하지 못한 사항도 포함한다.

## 완료 판정

요청 항목, 호환성 확인, 검증, 문서 갱신, 최종 상태 검토가 모두 끝났을 때만 완료다. 단순히 파일을 생성했거나 빌드가 한 번 통과한 것만으로 완료하지 않는다.
