# 디렉터리 문서 추적 규칙

## 목적

Codex가 파일을 변경하기 전에 책임과 기존 상태를 파악하고, 작업 후 다음 작업자가 검증 결과와 남은 문제를 이어받을 수 있도록 계층형 문서를 유지한다.

## 관리 대상

프로젝트가 직접 소유하고 사람이 변경하는 디렉터리는 `index.md`와 `progress.md`를 가진다. 중간 namespace 디렉터리도 현재 디렉터리 구조와 부모-자식 경계를 명확히 하기 위해 관리 대상에 포함한다.

이번 초기 기준 대상은 다음과 같다.

```text
./
├─ .codex/
│  └─ agents/
├─ .github/
│  └─ workflows/
├─ backend/
│  └─ src/main/
│     ├─ java/com/hiresemble/
│     └─ resources/db/migration/
├─ docs/
│  ├─ agent-rules/
│  └─ spec/
└─ frontend/
   ├─ e2e/
   └─ src/
      ├─ router/
      └─ styles/
```

표현을 압축한 경로의 모든 실제 중간 디렉터리(`backend/src`, `backend/src/main`, `backend/src/main/java`, `backend/src/main/java/com`, `backend/src/main/resources`, `backend/src/main/resources/db`)에도 두 문서를 둔다.

## 제외 대상

다음은 문서를 생성하지 않는다.

- VCS/IDE: `.git`, `.idea`, `.vscode`
- 의존성/외부 도구: `node_modules`, `.pnpm-store`, `vendor`, `backend/gradle`, Gradle Wrapper 내부
- 빌드/테스트 결과: `build`, `dist`, `target`, `out`, `coverage`, `playwright-report`, `test-results`, `blob-report`
- cache/temp: `.gradle`, `.vite`, `.cache`, `tmp`, `temp`, logs
- 자동 생성 코드/문서: `generated`, OpenAPI generator 출력 등
- 로컬 인프라 데이터: `infra/data`, `docker-data`

이름만으로 불명확하면 파일의 소유권과 생성 주체를 확인한다. 사람이 직접 관리하는 소스면 포함하고 도구가 재생성하면 제외한다.

## `index.md` 작성 규칙

현재 구조와 안정적인 책임을 설명한다. 최소 항목은 다음과 같다.

1. 디렉터리 목적
2. 주요 파일 및 하위 디렉터리
3. 각 구성 요소의 역할
4. 다른 디렉터리와의 의존 관계
5. 변경 시 주의사항
6. 관련 규칙 및 문서의 상대 경로 링크

새 파일을 모두 나열하지 말고 책임을 이해하는 데 필요한 주요 항목만 기록한다. 구조, 소유권, 공개 경계가 바뀔 때 갱신한다.

## `progress.md` 작성 규칙

모든 관리 대상 `progress.md`는 다음 구조를 사용한다.

```markdown
# Progress

## Overview

해당 디렉터리 또는 모듈의 진행 상황을 설명하는 짧은 개요

## [YYYY-MM-DD] Session Summary (구체적인 작업 제목)

- What was done:
  수행한 작업과 주요 변경 사항

- Key decisions:
  작업 과정에서 확정한 설계 및 기술적 결정

- Issues encountered:
  확인된 문제, 제약, 실패 또는 주의사항
  문제가 없으면 `None`으로 기록

- Validation:
  실행한 테스트, 빌드, 정적 분석 또는 검증 명령과 결과
  검증하지 않았다면 그 이유를 명시

- Next steps:
  남은 작업 또는 다음 권장 작업
```

날짜는 작업 시점의 한국 표준시를 기준으로 `YYYY-MM-DD`를 사용한다. Session Summary는 날짜 내림차순으로 정렬하며 새 기록은 `Overview` 바로 아래에 삽입한다. 같은 날짜에 여러 작업이 있으면 구체적인 제목을 가진 별도 Session Summary로 기록하고, 이미 같은 날짜인 기존 기록의 상대 순서는 근거 없이 바꾸지 않는다.

다음 작성 원칙을 적용한다.

- 작업 제목은 실제 수행 내용을 구분할 수 있게 쓰고 `작업 완료`, `수정 완료` 같은 표현만 사용하지 않는다.
- 핵심 파일과 변경 책임을 기록하되 수정 파일 전체 목록, 전체 빌드 로그와 스택 트레이스를 복사하지 않는다.
- 실패한 테스트, 미실행 검증, 미해결 문제를 숨기지 않고 실제로 실행하지 않은 테스트를 통과로 기록하지 않는다.
- 기존 Session Summary의 의미를 바꾸거나 새로운 사실을 만들지 않는다.
- 중간의 중복 `Overview`, `Notes`와 일반 상태 섹션은 의미를 보존해 상단 Overview 또는 해당 Session의 표준 필드로 옮긴다.
- 과거 기록을 삭제하거나 요약해 유실시키지 않는다. 오래된 세부가 많아도 키워드 기반 조회 규칙으로 컨텍스트를 제한하고 기록 자체는 보존한다.
- 날짜가 섞인 기존 기록은 날짜 내림차순으로 정렬한다.
- 중요하지 않은 단순 포맷 또는 오탈자 수정만 수행했다면 불필요한 Session Summary를 만들지 않는다.

## `progress.md` 조회 규칙

기본 조회에서 작업 기록으로 세는 제목은 다음 정규 패턴에 맞는 항목뿐이다.

```text
## [YYYY-MM-DD] Session Summary (...)
```

`Overview`, `Notes`와 일반 제목은 개수에 포함하지 않는다. 동일 날짜의 여러 Session Summary는 각각 한 개이며, 최신순 문서의 첫 5개만 읽는다. 기록이 5개 미만이면 존재하는 기록을 모두 읽는다.

- 루트 관리자는 루트, 실제 작업 최상위 모듈과 직접 관련 디렉터리의 최신 5개만 읽는다.
- 전문 서브 에이전트는 할당 디렉터리와 필요할 때 가장 가까운 상위 관리 디렉터리의 최신 5개만 읽는다.
- 관련 없는 디렉터리 또는 루트부터 모든 하위 기록을 연쇄 조회하지 않는다.
- 과거 결정 원인, 동일 오류, 특정 기능·파일 이력 또는 회귀 원인을 조사할 때만 작업 키워드, 파일명, 클래스명, 오류 메시지, 기능명 또는 날짜 범위로 제한해 추가 검색한다.
- 추가 조회 범위와 이유는 작업 결과에 남긴다.

## 갱신 책임

- 루트 전체에 영향을 주는 변경은 루트 `progress.md`에 전체 영향과 통합 결과를 기록한다.
- 한 모듈에만 영향을 주면 해당 모듈 기록에 구체적인 변경과 검증을 남긴다.
- 여러 모듈에 영향을 주면 각 모듈에는 영역별 세부를, 루트에는 중복하지 않은 통합 요약을 남긴다.
- 디렉터리 구조나 책임이 변경되면 해당 `index.md`도 갱신한다.
- 서브 에이전트는 구조화 결과와 문서 갱신 필요 사항을 루트 관리자에게 반환하며 `index.md`와 `progress.md`를 직접 수정하지 않는다.
- 최종 추적 문서 갱신은 루트 관리자가 통합해 수행한다.

## 계층 원칙

- 루트 `index.md`: 전체 프로젝트 구조와 모듈 관계
- 루트 `progress.md`: 여러 모듈에 걸친 전체 상태와 결정
- 모듈 문서: 모듈의 공개 책임, 내부 영역, 모듈 단위 검증
- 하위 문서: 해당 package/기능/설정 영역의 세부 상태

상위 문서에는 하위 구현 목록을 반복하지 않고 링크와 영향만 기록한다. 하위 문서에는 프로젝트 전체 소개를 반복하지 않는다.

## 변경 전후 절차

변경 전에는 역할에 필요한 `index.md`와 위 최신 5개 범위의 `progress.md`를 읽는다. 변경 후에는 직접 영향받은 디렉터리 문서를 갱신한다. 파일을 다른 디렉터리로 이동하면 출발지와 도착지 문서를 모두 갱신한다. 프로젝트 전체 규칙, 공통 계약, root 설정 변경은 루트 `progress.md`에도 요약한다. 서브 에이전트를 사용한 경우 루트 관리자가 모든 결과를 받은 뒤 이 갱신을 수행한다.

빌드나 테스트 실패는 원인과 미검증 범위를 남긴다. 문서 갱신만으로 상태를 “구현 완료”로 올리지 않는다.

## 링크 규칙

- 저장소 내부 문서는 상대 경로 Markdown 링크를 사용한다.
- 링크는 실제 파일 또는 디렉터리를 가리켜야 한다.
- 같은 파일을 상·하위 문서에서 장문으로 복제하지 않는다.
- 이동 시 inbound link를 `rg`로 찾아 함께 고친다.
